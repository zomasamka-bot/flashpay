           │
           ├─ Lock ACQUIRED ✓
           │  ├─ Reload latest payment checkpoint
           │  ├─ Check skip flags (a2uTxid, horizonSuccessFlag)
           │  ├─ Call executeA2U() [ONLY inside lock]
           │  ├─ Stages: 0 (check), 1 (create), 2 (sign, conditional skip), 3 (Pi), 4 (DB)
           │  └─ Return result
           │
           └─ Lock FAILED ✗
              ├─ Re-read current payment state from Redis
              ├─ DO NOT execute executeA2U()
              └─ Return current state (authoritative)
           ↓
        Call buildA2USuccessResponse() [outside lock]
           ├─ Read Redis checkpoint
           ├─ Validate predicate (settled_to_merchant status required)
           └─ Return final response
           ↓
        HTTP Response

================================================================================
LOCK SPECIFICATION
================================================================================

Key Format:         a2u:lock:${paymentId}
Lock Strategy:      redis.set(key, token, { nx: true, ex: ttl })
TTL:                600 seconds (10 minutes)
Token:              crypto.randomUUID() — unique per acquisition
Token Verification: Lua script checks token matches before DELETE
Expiry Behavior:    Automatic unlock after 10 minutes (crash protection)

Lock Failure Flow:
  If lock not acquired (another caller holds it):
  └─ Re-read current payment state from Redis
  └─ Return current authoritative checkpoint
  └─ Do NOT execute any A2U stages

Stage 2 Skip Conditions (inside lock):
  if (payment.a2uTxid || payment.horizonSuccessFlag) {
    └─ Skip Horizon signing permanently (Stage 2)
    └─ Executor reuses existing txid or skips to Stage 3
  }

Atomic Release (finally block):
  Lua script: if redis.get(lockKey) == token then redis.del(lockKey)
  └─ Prevents wrong caller from releasing lock
  └─ Preserves lock if token mismatch (concurrent caller holds it)
